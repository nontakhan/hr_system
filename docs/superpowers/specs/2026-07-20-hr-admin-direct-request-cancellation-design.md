# การยกเลิกคำขอพนักงานโดย HR/Admin

## เป้าหมาย

ให้ผู้ใช้บทบาท HR และ Admin สามารถยกเลิกคำขอที่อนุมัติแล้วของพนักงานภายในขอบเขตที่ตนมีสิทธิ์ดูแลได้ทันที โดยต้องระบุเหตุผลและมีหลักฐานตรวจสอบย้อนหลังครบถ้วน

## ขอบเขตคำขอ

- การลา
- มาสายและออกก่อน
- ทำงานล่วงเวลา
- สลับวันหยุด
- อบรมหรือกิจกรรม

## ตำแหน่งใน UI

เพิ่มปุ่ม `ยกเลิกรายการ` ในแท็บ `ประวัติการอนุมัติ` ของหน้าการอนุมัติแต่ละประเภท ปุ่มแสดงเฉพาะแถวที่มีสถานะ `approved` และผู้ใช้ปัจจุบันเป็น HR หรือ Admin ที่มีสิทธิ์ตาม HR scope ของพนักงานเจ้าของคำขอ

ปุ่มใช้สีแดง ขนาดกะทัดรัด และวางในคอลัมน์การดำเนินการหรือบริเวณสถานะตามโครงสร้างเดิมของแต่ละตาราง โดยไม่แสดงในหน้าประวัติส่วนตัวของพนักงาน

## ขั้นตอนยกเลิก

1. HR/Admin กด `ยกเลิกรายการ`
2. ระบบแสดงกล่องยืนยันที่ระบุชื่อพนักงาน ประเภทคำขอ และวันที่ที่เกี่ยวข้อง
3. ผู้ใช้ต้องกรอกเหตุผลการยกเลิก โดยตัดช่องว่างหัวท้ายและไม่ยอมรับข้อความว่าง
4. เมื่อยืนยัน API ตรวจ session, role, HR scope, request id และสถานะปัจจุบันอีกครั้ง
5. ระบบเปลี่ยนสถานะจาก `approved` เป็น `cancelled` ทันทีด้วยเงื่อนไข atomic `WHERE id = ? AND status = 'approved'`
6. UI แจ้งผลและโหลดตารางประวัติใหม่

ไม่ส่งรายการเข้าคิว `pending_cancel_hr` และไม่ต้องผ่านการอนุมัติซ้ำ

## สิทธิ์และ HR Scope

- อนุญาตเฉพาะบทบาท `hr` และ `admin`
- ใช้ HR scope helper เดิมกับ employee id ของคำขอทุกครั้งที่อ่านและเขียน
- การซ่อนปุ่มใน UI เป็นเพียงการช่วยใช้งาน API ต้องเป็นจุดบังคับสิทธิ์หลัก
- Manager และพนักงานทั่วไปเรียก action นี้ไม่ได้

## Audit และข้อมูล

เพิ่มข้อมูลการยกเลิกแยกจากข้อมูลการอนุมัติเดิมในตารางคำขอที่เกี่ยวข้อง:

- `cancelled_by_user_id`
- `cancelled_by_employee_id`
- `cancelled_by_role`
- `cancelled_at`
- ใช้ `cancellation_reason` เดิมสำหรับเหตุผล

ห้ามเขียนทับ `approver_id`, `approval_date`, `manager_approver_id`, `manager_approval_date`, `hr_approver_id` หรือ `hr_approval_date` เพื่อคงประวัติว่าใครเป็นผู้อนุมัติเดิม

หน้าประวัติแสดงเหตุผล ผู้ยกเลิก บทบาท และวันเวลายกเลิกสำหรับรายการที่ถูกยกเลิกโดย HR/Admin หากเป็นรายการยกเลิกเดิมที่ไม่มี audit fields ให้แสดงข้อมูลเท่าที่มีโดยไม่สร้างชื่อผู้ดำเนินการขึ้นเอง

## ความสอดคล้องของข้อมูล

- ใช้ transaction เมื่อต้องปรับข้อมูลประกอบหลายรายการ
- ตรวจ `affected_rows === 1` เพื่อป้องกันการยกเลิกซ้ำหรือสถานะเปลี่ยนระหว่างดำเนินการ
- การยกเลิกยังคงไหลเข้าสู่ attendance, quota, calendar และ report consumers ผ่านสถานะ `cancelled` ตาม semantics เดิม
- ไม่ลบแถวคำขอและไม่ลบประวัติการอนุมัติ

## API

ใช้ action สำหรับ reviewer cancellation แยกจาก employee self-cancellation เพื่อไม่ให้เงื่อนไขเจ้าของคำขอและ `pending_cancel_hr` ปะปนกัน แต่ละ endpoint รับเฉพาะ:

- `action`
- `request_id`
- `cancellation_reason`
- ตัวจำแนกประเภทคำขอที่ endpoint ร่วมจำเป็นต้องใช้ เช่นมาสาย/ออกก่อนกับ OT

ไม่รับ employee id, role หรือชื่อผู้ยกเลิกจาก client แต่ดึงจากคำขอและ session ฝั่ง server เท่านั้น

## Error Handling

- เหตุผลว่าง: แจ้งให้ระบุเหตุผล
- ไม่มีสิทธิ์หรืออยู่นอก HR scope: ตอบปฏิเสธโดยไม่เปิดเผยข้อมูลคำขอ
- ไม่พบรายการหรือสถานะไม่ใช่ `approved`: แจ้งว่าสถานะเปลี่ยนแล้วและให้โหลดข้อมูลใหม่
- บันทึกไม่สำเร็จ: rollback และคงสถานะเดิม

## การทดสอบ

- Contract/regression tests ครบทั้งห้าประเภท
- ทดสอบ role HR/Admin สำเร็จ และ manager/employee ถูกปฏิเสธ
- ทดสอบ HR scope ทั้งกรณีในขอบเขตและนอกขอบเขต
- ทดสอบเหตุผลว่าง, request id ไม่ถูกต้อง, สถานะไม่ใช่ approved และการเรียกซ้ำ
- ยืนยันว่าข้อมูลผู้อนุมัติเดิมไม่ถูกเขียนทับ
- ยืนยันว่า UI แสดงปุ่มเฉพาะ approved และแสดง audit หลังยกเลิก
- ตรวจ syntax PHP/JavaScript, focused suites และ `git diff --check`
- ตรวจ workflow จริงผ่าน Apache/XAMPP เมื่อฐานข้อมูลพร้อม

## เกณฑ์ยอมรับ

- HR/Admin ยกเลิกคำขอ approved ของพนักงานใน scope ได้ทันทีพร้อมเหตุผล
- ผู้ใช้นอกบทบาทหรือนอก scope ทำไม่ได้ทั้งจาก UI และ API
- ประวัติแสดงผู้ดำเนินการ เหตุผล และเวลาอย่างตรวจสอบย้อนหลังได้
- ข้อมูลผู้อนุมัติเดิมยังอยู่ครบ
- พฤติกรรม employee self-cancellation เดิมไม่เปลี่ยน
